# 对抗审查

## 核心机制
不要只从一个视角看自己的设计。依次切换以下四个角色，每个角色独立给出批评。

## 审查角色

### 1. 目标受众
- 这页我能一眼看懂核心观点吗？
- 有没有让我困惑的术语或数据？
- 这页在帮我达到演示目标吗？

### 2. 挑剔设计师
- 视觉是否统一？配色、字体、留白是否协调？
- 有没有过度装饰？
- 信息层级是否清晰？

### 3. 逻辑审查者
- 这一页和上一页是否连贯？
- 论据是否支撑观点？
- 有没有冗余或遗漏？

### 4. 时间管理者
- 这页在总时长里占多少？
- 有没有某页信息过载？
- 结尾是否有明确的行动号召？

## 量化评分（0-100 分）

每个维度 0-25 分，总分决定是否需要迭代：

| 维度 | 分值 | 检查点 |
|---|---|---|
| **内容** (25) | 核心观点是否一页一个？论据是否支撑结论？是否有冗余？ |
| **叙事** (25) | 前后页是否连贯？金字塔结构是否清晰？开场是否抓人？结尾是否有 CTA？ |
| **视觉** (25) | 配色是否统一？dominance 是否明确？母题是否贯穿？对比度是否足够？ |
| **密度** (25) | 每页信息是否适中？文字是否精简？图表是否必要？ |

**总分门槛**：
- **≥ 80 分** 且无 High 问题 → 通过，可交付
- **60-79 分** 或存在 High 问题 → 必须迭代，修复 weakest 3 项
- **< 60 分** → 大改，回到阶段 2（信息架构）重新审视

## 输出格式
```markdown
## 审查报告

### 量化评分
- 内容：X/25
- 叙事：X/25
- 视觉：X/25
- 密度：X/25
- **总分：X/100**

### High（必须修复）
1. [角色] [问题] [修改建议]

### Medium（建议修复）
1. ...

### Low（可选）
1. ...
```

## 评分规则
- **< 80 分** 或任一 High 问题未修复 → 必须迭代
- 迭代时只修复评分最低的 3 个维度，不要借机会重写整个 deck
- 无 High、总分 ≥ 80 且 Medium ≤ 3 → 可以通过
